# 15. Блок «Бурение»: этап 1 — проектирование

## 1) Цель этапа

Собрать цифровой контур бурения, в котором инженерные решения по режимам и рискам принимаются на основе данных и AI-рекомендаций в режиме, близком к реальному времени.

## 2) Бизнес-результат этапа 1

- Снижение NPT на 10-15% на пилотном периметре
- Рост ROP на 5-12% без нарушения ограничений безопасности
- Сокращение времени инженерного цикла принятия решений на 20-40%

## 3) Периметр пилота

- 1-2 буровые бригады
- 5-15 скважин (или интервалов) для пилотного цикла
- Историческое окно данных: 6-12 месяцев
- Горизонт проектирования: 90 дней

## 4) Архитектура блока «Бурение"

### Источники данных

- WITS/WITSML поток
- Буровые суточные отчеты
- Геолого-технические наряды
- Событийные журналы осложнений
- Данные по долотам и КНБК

### Обработка и качество данных

- Нормализация единиц измерения
- Синхронизация временных рядов
- Флаги пропусков, выбросов, задержек
- Data quality score по каждой скважине

### AI-модули этапа 1

- NPT Predictor
- ROP Advisor
- Stuck Pipe Early Warning
- Drift Detection по ключевым параметрам режима

## 5) Решения и workflows

1. Поток телеметрии ->
2. AI-оценка риска и рекомендации ->
3. Подтверждение супервайзером (human-in-the-loop) ->
4. Выполнение действия ->
5. Фиксация эффекта и обратной связи в модели

## 6) KPI блока «Бурение"

- NPT Reduction (%)
- ROP Improvement (%)
- Precision/Recall алертов по осложнениям
- Recommendation Adoption Rate (%)
- Time-to-Decision (мин)

## 7) Безопасность и ограничения

- Рекомендации не должны обходить HSE-барьеры
- Критические действия только после подтверждения инженером
- Автоматическое отклонение рекомендаций вне допустимого технологического окна

## 8) План проектирования на 6 недель

### Недели 1-2

- Определение периметра, KPI и baseline
- Инвентаризация и подключение данных
- Описание критериев качества данных

### Недели 3-4

- Проектирование признаков и baseline-моделей
- Сценарии принятия решений в командном центре
- Проработка интерфейса алертов для супервайзера

### Недели 5-6

- Тестирование shadow mode
- Калибровка порогов алертов
- Подготовка к ограниченному production запуску

## 9) Команда этапа

- Product Owner (бурение)
- Domain Engineer (бурение)
- Data Engineer
- ML Engineer
- MLOps Engineer
- Remote Ops Engineer

## 10) Критерии готовности к этапу 2

- Подтвержденный эффект минимум по 2 KPI
- Качество данных >= согласованного порога
- SLA по алертам и отчетности выполняется
- Заказчик подтверждает расширение периметра
